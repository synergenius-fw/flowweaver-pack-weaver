# Weaver Pack — Vision & Spec

## What This Is

Weaver is the autonomous AI bot system for Flow Weaver. It has three layers:

1. **Assistant** — conversational AI that manages bots, queues, and the flow-weaver ecosystem
2. **Bot Sessions** — separate processes that execute tasks via tool-use agent loops
3. **Tool Layer** — validate, read, patch, shell, git — with safety guards

The assistant is the director. Bots are the workers. The user talks to the assistant, the assistant orchestrates bots.

## Architecture Constraints

- Bots are SEPARATE PROCESSES — they survive assistant restart
- All inter-process communication is FILE-BASED (queue NDJSON, steering JSON, output logs)
- No external dependencies beyond Node.js stdlib + flow-weaver
- All output is PIPE-SAFE (decorative on stderr, structured on stdout)
- The tool layer is SHARED between bots and the assistant (same safety guards)
- The assistant core is UI-AGNOSTIC — terminal CLI is one frontend, platform AI chat is another

## Quality Standards

- TDD: write failing tests FIRST, then implement
- Every new file needs tests — target 90%+ coverage
- Every bug fix needs a regression test
- No shortcuts, no workarounds — fix root causes
- No console.log in production code — use TerminalRenderer
- Enterprise-grade error handling: every catch block logs meaningful context
- TypeScript strict mode, no `any` types

## Code Patterns

- Node types are pure functions with JSDoc annotations — no side effects in imports
- Async operations use withFileLock for concurrency safety
- NDJSON for append-only stores (queue, costs, audit, messages)
- JSON for single-record state (session, steering, conversation index)
- All CLI flags parsed in parseArgs(), dispatched via cli-bridge.ts handler map

## What Bots Must Do

1. Read the task instruction carefully
2. Use tools to discover the current state (validate, read_file, list_files)
3. Make targeted changes (patch_file preferred over write_file)
4. Validate after every change
5. Only commit when validation passes (post-agent validation gate enforces this)
6. Never commit to main on overnight runs — use a branch

## What Bots Must NOT Do

- Delete files without explicit instruction
- Run destructive shell commands (rm -rf, git push, npm publish — blocked by safety guards)
- Modify files outside the project directory (path traversal blocked)
- Write empty content or truncate files >50% (shrink guard blocks this)
- Skip validation after patching
- Commit code that doesn't compile
- Add dependencies without explicit instruction
- Change the architecture without being told to

## Current Priorities (update this section as needed)

1. Test coverage — every file in src/bot/ and src/node-types/ needs comprehensive tests
2. Reliability — retry logic, error recovery, graceful degradation
3. UX — clean terminal output via TerminalRenderer, no raw ANSI in node-types
4. Documentation — JSDoc on all public functions, help text for all CLI commands

## Autonomous Improvement Rules (for `weaver improve`)

When running autonomously, the assistant MUST:
- Only work on items from "Current Priorities" above
- Start with test coverage (priority 1) before moving to reliability (priority 2)
- Write failing tests FIRST, then implement the fix
- Keep each change to 1-3 files maximum
- Never modify: assistant-core.ts, tool-registry.ts, cli-handlers.ts, cli-bridge.ts, improve-loop.ts, workflow files, UI files, config files
- Never add dependencies, change architecture, or do cosmetic refactoring
- Stop if unsure — skip the cycle rather than make a questionable change

## Anti-Patterns to Avoid

- Don't refactor code you're not tasked to change
- Don't add features beyond what's asked
- Don't add comments explaining obvious code
- Don't create utility files for one-time operations
- Don't wrap existing functions in unnecessary abstractions
- Don't add backward-compatibility shims — just change the code
- Don't use emoji in output or comments

## File Structure

```
src/
  bot/                    # Core bot infrastructure
    assistant-core.ts     # Conversational loop
    assistant-tools.ts    # Tool definitions + executor (21 tools)
    bot-manager.ts        # Process manager for bot fleet
    conversation-store.ts # Persistent conversation history
    task-queue.ts         # NDJSON task queue with file locking
    steering.ts           # File-based control commands
    session-state.ts      # Session state persistence
    terminal-renderer.ts  # Centralized output (icons, colors, timing)
    retry-utils.ts        # Transient error detection + exponential backoff
    cost-tracker.ts       # Token usage and cost estimation
    system-prompt.ts      # AI system prompt builder
    step-executor.ts      # Tool execution with safety guards
    weaver-tools.ts       # Bot tool definitions (validate, read, patch, shell)
  node-types/             # Flow Weaver node types (pure functions)
  workflows/              # Compiled workflow files
  cli-handlers.ts         # CLI command handlers
  cli-bridge.ts           # Command dispatch map
```

## Testing Strategy

- Unit tests per file in tests/ directory
- Mocked dependencies (vi.mock for fs, child_process)
- Real file system tests use os.tmpdir() with cleanup
- Integration tests verify CLI routing end-to-end
- Full suite must pass before any commit
